Chrome 报 SSL 证书错误？可能是代理工具干的

2026年6月14日 · 约 8 分钟阅读 · 分类：系统管理 / 网络调试

前几天装了 WattToolkit（也就是以前的 Steam++），用来加速访问 GitHub。装完之后发现 Chrome 访问任何 HTTPS 网站都报 ERR_CERT_AUTHORITY_INVALID 错误。

一开始以为是系统证书出了问题，折腾了半天才发现，罪魁祸首其实是 WattToolkit 的中间人代理机制。记录一下排查过程和修复方法，说不定有人会遇到同样的问题。

问题现象

Chrome 打开 https://github.com，报错：

您的连接不是私密连接
攻击者企图从文件中窃取您的信息（github.com）。您的计算机信任的证书颁发机构没有颁发认可此网站安全连接的证书。

错误代码：ERR_CERT_AUTHORITY_INVALID

但用 curl 访问也是同样的错误——说明不是 Chrome 单独的问题，是整个系统的 SSL 证书信任链断了。

排查过程

第一步：确认是不是代理的问题

我先关了 WattToolkit 的代理，然后刷新页面——Chrome 正常了。说明问题就在代理这边。

WattToolkit 的原理是在本地起一个反向代理（监听 443 端口），对所有 HTTPS 流量做中间人解密。它会用自己的根证书签发中间人证书，然后让浏览器信任这个证书。

问题是——它的根证书没有正确安装到系统的证书信任库中。

第二步：检查系统证书

查看 /etc/ssl/certs/ca-certificates.crt 这个文件，发现里面只有 1 个证书。正常情况下这个文件应该有 140+ 个系统根证书。这说明有人（可能是我之前做实验）把系统证书替换掉了。

# 查看系统证书数量
grep -c "BEGIN CERTIFICATE" /etc/ssl/certs/ca-certificates.crt
# 输出：1（不正常！应该是 140+）

# 正常情况下的输出示例
grep -c "BEGIN CERTIFICATE" /etc/ssl/certs/ca-certificates.crt
# 输出：147

第三步：找到 WattToolkit 的证书

WattToolkit 的根证书在：~/.local/share/Steam++/Plugins/Accelerator/SteamTools.Certificate.cer

这个证书需要安装到两个地方：

  1. 系统证书库——让 curl、wget 等命令行工具信任
  2. NSS 数据库——让 Chrome/Chromium 信任（Linux 上 Chrome 用 NSS 管理证书，不用系统证书库）

修复方案

修复1：还原系统证书

先把被替换的系统证书还原回去：

# 从 certifi 包提取所有根证书
python3 -c "
import certifi
with open(certifi.where(), 'r') as f:
    print(f.read(), end='')
" > /tmp/ca-bundle.crt

# 备份旧证书
sudo cp /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/ca-certificates.crt.bak

# 替换为正确的证书
sudo cp /tmp/ca-bundle.crt /etc/ssl/certs/ca-certificates.crt

# 更新系统证书
sudo update-ca-certificates

验证一下：

grep -c "BEGIN CERTIFICATE" /etc/ssl/certs/ca-certificates.crt
# 输出：147 ✅

修复2：配置 Chrome NSS 信任

Linux 上的 Chrome 不使用系统证书库，而是使用自己的 NSS 数据库。所以需要把 WattToolkit 的根证书添加到 NSS 数据库中，并设置信任标志。

# 找到 NSS 数据库位置
NSS_DIR="$HOME/.pki/nssdb"

# 导入 WattToolkit 根证书
certutil -A -n "SteamTools CA" \
  -t "CT,C,C" \
  -i ~/.local/share/Steam++/Plugins/Accelerator/SteamTools.Certificate.cer \
  -d sql:$NSS_DIR

# 验证证书已导入
certutil -L -d sql:$NSS_DIR

这里的信任标志 CT,C,C 意思是：

  • C（Trusted Certificate）—— 信任为根证书颁发机构
  • T（SSL）—— 信任用于 SSL 服务端认证
  • C（SMTP）—— 信任用于邮件签名（这里设为空信任）

修复3：重启 Chrome

做完以上两步之后，关掉 Chrome 再重新打开。访问 GitHub，错误消失了。

为什么会发生这个问题？

根本原因是 WattToolkit 的 YARP 反向代理在尝试签发中间人证书时，需要引用系统根证书来构建完整的证书链。但当时系统的 /etc/ssl/certs/ca-certificates.crt 被我之前的操作替换成了只有单个证书的文件，导致 WattToolkit 找不到完整的证书链，签发的证书不被信任。

WattToolkit 的日志里也明确报了错：

System.Security.Cryptography.CryptographicException: 
OpenSSL mapping function MapOpenSSL30Code reported 
unhandled error code '30'.
// error code 30 = X509_V_ERR_UNABLE_TO_GET_ISSUER_CERT_LOCALLY

错误码 30 的意思是："无法在本地获取颁发者证书"——也就是证书链断了。

如何避免类似问题？

  1. 不要轻易替换系统证书文件。/etc/ssl/certs/ca-certificates.crt 是系统级的，动错了会影响所有 HTTPS 连接。
  2. 代理工具出问题先关代理。Chrome 报 SSL 错误时，先关掉所有代理工具，看看是不是代理引起的。
  3. Linux 上 Chrome 和系统证书是分开的。Chrome 用 NSS 数据库，不直接读系统证书。装了代理工具后，可能需要同时在两个地方配置信任。
  4. 定期备份证书文件。sudo cp /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/ca-certificates.crt.bak，养成好习惯。

总结

这个问题折腾了我不少时间，主要是在排查过程中走了弯路——一开始以为系统证书坏了，修了半天才发现真正的问题是 Chrome 的 NSS 数据库没有信任 WattToolkit 的证书。

最后总结下来就两件事：

  1. 还原系统根证书（147 个）
  2. 在 Chrome NSS 数据库中信任 WattToolkit 的根证书

希望这篇记录能帮到遇到同样问题的同学。如果你的 Chrome 也突然报 ERR_CERT_AUTHORITY_INVALID，不妨检查一下是不是装了什么代理工具。
