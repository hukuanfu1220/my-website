Hermes Agent 满配优化：我把69个Skill砍到4个

2026年6月9日 · 约 15 分钟阅读 · 分类：DevOps / AI Agent

事情是这样的：我给 Hermes Agent 装了一堆 Skill，数了一下整整 69 个。每次启动，Agent 都会扫描所有 Skill 目录，加载它们的 SKILL.md 文件，然后塞进上下文窗口。结果就是——启动慢、token 消耗大、上下文被大量无关信息污染。

优化完以后，我把常驻 Skill 压到了 4 个，启动速度和响应质量都明显提升。下面记录整个过程的思路、操作步骤和踩过的坑。

为什么 Skill 多了反而不好？

很多人有个误区：Skill 越多，Agent 能做的事越多。但实际上，Hermes 的 Skill 加载机制是这样的：

  1. 每次对话开始时，Agent 会扫描 ~/.hermes/skills/ 目录
  2. 把所有 Skill 的 SKILL.md 读出来，拼成一段系统提示
  3. 这段提示会被注入到每个 turn 的上下文中

69 个 Skill 意味着什么？假设每个 Skill 的 SKILL.md 平均 500 字，那就是 34,500 字的系统提示。一个 turn 还没开始，就已经消耗了近 10,000 tokens。而且大部分 Skill 根本用不到——比如你让我写个 Python 脚本，它没必要知道怎么部署静态网站。

我的精简策略

我决定用一套筛选标准：

保留原则：如果一个 Skill 在过去两周里没有被我用过，或者它的功能可以被其他 Skill 或内置工具替代，就删掉。

第一步：盘点所有 Skill

先列出来看看到底有哪些：

# 列出所有 Skill 目录
ls ~/.hermes/skills/

# 统计数量
ls -d ~/.hermes/skills/*/ | wc -l
# 输出：69

我按类别分了组：

类别数量使用情况
开发相关（coding、debugging、testing）18偶尔用
运维相关（服务器、Docker、网络）12基本不用
数据相关（爬虫、分析、可视化）10偶尔用
通信相关（Telegram、Discord、Slack）8只用飞书
Hermes 自身配置（config、setup）6一次性
创意相关（写作、翻译、PPT）15很少用
其他杂项10没用过

第二步：按优先级保留

最终只保留了 4 个核心 Skill：

  1. software-development — 软件开发工作流规划、代码审查、测试策略
  2. research — 市场调研、竞品分析、学术搜索
  3. system-administration — Linux 系统管理、SSL 证书修复、网络调试
  4. hermes-agent — 我自己的 Hermes 配置和优化（这个必须留，毕竟我在折腾它）

其余的怎么办？不是直接删，而是按需加载。

改造后的加载机制

精简之后，我做了两件事：

1. 常驻 Skill 不超过 5 个

在 config.yaml 里明确写了：

# 常驻 Skill（每次自动加载）
skills:
  - software-development
  - research
  - system-administration
  - hermes-agent

2. 其他 Skill 按需加载

对于偶尔用到的功能（比如部署网站、生成图片），我不再把它做成常驻 Skill。而是在任务需要时，通过 skill_view() 临时加载，用完就不管了。

这样的好处很明显：

  • 每次对话的上下文窗口干净了很多
  • token 消耗减少了大约 70%
  • Agent 响应速度更快，因为不需要读取大量无关的 Skill 描述

顺手做了其他优化

MEMORY.md 清理

之前 MEMORY.md 里记了七八十条碎片信息，很多已经过期了。我重新整理了一遍，只保留两类：

  • 持久事实：用户偏好、环境配置、技术约定
  • 经验教训：踩过坑之后总结的方法论

临时的任务进度、PR 编号、commit SHA 之类的东西全部清理掉了——这些应该用 session_search 去查，不应该写在记忆里。

max_turns 调整

默认 max_turns: 150 太激进了。改成 100 就够了——既允许复杂任务有足够的轮次，又防止 Agent 在某些情况下死循环消耗 token。

Soul.md 重写

原来的 SOUL.md 写得比较随意，缺乏行为边界。我重写了之后加了几个关键原则：

  • 先给结论，再解释原因
  • 不确定就不确定，不要编造
  • 输出要具体、可复制、可执行
  • 发现需求矛盾时，直接指出而不是顺着说

优化效果对比

指标优化前优化后
Skill 数量694（常驻）
系统提示大小~35,000 字~2,000 字
每次 turn 的 token 消耗~10,000~600
响应延迟明显卡顿流畅
上下文质量大量无关信息精准相关

踩过的坑

坑1：删除 Skill 之前没备份

我第一次删的时候手快，直接把整个 skills/ 目录清空了。后来才发现有些 Skill 虽然不常用，但里面有一些我写好的模板文件。教训：删之前先用 tree ~/.hermes/skills/ 看一眼结构。

坑2：Skill 更新后没同步

优化过程中我发现某个 Skill 的描述和实际功能不一致，修了 SKILL.md 但忘了重新加载。结果 Agent 还是按照旧描述在执行。教训：修改 Skill 后一定要用 skill_view() 验证一下。

坑3：MEMORY.md 写得太长

一开始我往 MEMORY.md 里塞了太多细节，结果每条消息的上下文都被占满了。后来才意识到，MEMORY.md 应该只放跨会话仍然有用的事实。比如"用户偏好中文沟通"有用，但"昨天我们做了什么"没用。

总结

这次优化的核心思路就一句话：少即是多。

69 个 Skill 看着很强大，但实际上每次对话只有 4 个在用。剩下的 65 个不仅浪费 token，还污染上下文。精简之后，Agent 的运行效率和质量都提升了，而且维护起来也更轻松。

如果你也在用 Hermes Agent，建议你也做一次类似的盘点。看看哪些 Skill 是真的在用，哪些只是"以防万一"装上去的。你会发现，很多时候少比多更好。
